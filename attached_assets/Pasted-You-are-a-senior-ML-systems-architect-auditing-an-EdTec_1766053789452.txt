	•	If something is implied but not present, label it: Planned / Not Implemented.
	•	For every model/algorithm: identify the exact function/class name, file path, and where it is called.
	•	For every stored artifact: identify the exact table/model name, field names (if visible), and where reads/writes happen.

Output: one Markdown document with these sections

1) Model & Algorithm Inventory (Verified Only)

Create a table with one row per model/algorithm. Columns:
	•	Component Name
	•	Type (LLM, bandit, rules engine, scoring model, classifier, retrieval, scheduler, etc.)
	•	Algorithm (e.g., epsilon-greedy, rolling average, weighted win-rate, rubric scoring, mastery update rule, etc.)
	•	Implementation Location (repo path + class/function)
	•	Inputs (runtime) (exact variables/fields)
	•	Outputs (runtime) (exact variables/fields)
	•	Persistence (what is stored, where)
	•	Consumers (which other components read those outputs)

You MUST include any of the following if present:
	•	Student Chat Tutor (subject-locked LLM)
	•	Teacher Assistant (LLM)
	•	Mastery/Skill Model (student skill estimates)
	•	Strategy Selector (epsilon-greedy bandit)
	•	Misconception Diagnosis (rule/LLM-based tagging + remediation)
	•	Analytics (heatmap / at-risk logic)
	•	Understanding model (profile interpretation / learning-state)
	•	Test/Quiz generator

2) Data Artifacts and Storage (Canonical Data Dictionary)

List every persisted data artifact as a table with columns:
	•	Artifact / Entity
	•	Storage Layer (SQLAlchemy table, SQLite/Postgres, file, cache, etc.)
	•	Schema Fields (field names + types if available)
	•	Written By (component + code path)
	•	Read By (component + code path)
	•	Lifecycle (created/updated/append-only)
	•	Retention (if implemented; else “not implemented”)

3) End-to-End Data Lineage Graph (Producer → Store → Consumer)

Produce a DAG-style mapping. For each edge, specify:

Producer component
→ Stored artifact (table + fields)
→ Consumer component
→ How it is used (prompt context, feature value, ranking signal, gating, remediation, analytics, etc.)

Use this explicit format repeatedly:
	•	Producer: <component>
Writes: <table.field1, table.field2...> (why)
Consumer: <component>
Reads: <table.field...> (how used)

Must cover at least:
	•	Chat message → AIInteraction/ChatMessage storage → profile summaries → next prompt assembly
	•	Quiz/quick check → scoring → mastery update → strategy reward update → next strategy selection
	•	Incorrect answers → misconception tags/remediation → teacher analytics + student tutor follow-up prompts

4) Algorithm Details (Explain Exactly What the Code Does)

For each non-LLM algorithm (bandit/mastery/scoring/thresholding):

Example structure (fill from code, don’t invent):

5) Prompt/Data Coupling (How Stored Data Feeds Models)

Describe how stored data is injected into each model:

6) Call-Site Index (Where Each Model Runs)

List all endpoints/jobs that invoke models, in this format:

List the biggest risks you find, e.g.:
